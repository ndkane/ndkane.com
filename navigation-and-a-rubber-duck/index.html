Fixing last week’s problems…

Last week I mentioned problems with the margins between my list items in unordered and ordered lists. I wanted to change them as any list item that were longer than one line just looked cluttered. I’d been making the mistake of adding the margins onto the lists themselves, rather than the items. This was easily fixed by tweaking the <li> element. I’ve been learning to use Safari’s Web Inspector more and more, rather than just guessing!

I also wanted to solve the padding on tables, as before nothing that I applied in the CSS would change the top padding. This I talked through with Aral, after inspecting the elements. After a bit of experimentation we found that anything I applied to <tr> would have no effect. Only padding on <td> would apply! Problem solved. While we were looking at tables, we came across pseudo-classes.

Pseudo-Classes, or How to Organise Your Children

My tables needed a bit of jazzing up. Keeping them in a block light-grey didn’t really look very nice, and wasn’t very clear. Therefore, I needed to look at pseudo-classes so that I could apply alternating colours to the cells. There are several pseudo-classes, in fact I’d previously used links-related pseudo-class selectors in order to highlight my links. For this particular example I needed to look at Position and Number-based pseudo-class selectors. There’s a great sheet of these over on CSS Tricks which I’ve found helpful as a reference.

Here’s the code as it appears in my stylesheet:

tr
{
  background-color: #CDCDCD;
}

tr:nth-child(2n)
{
  background-color: darkgrey;
}

tr:not(:last-child)
{
  border-bottom: 1px solid darkgrey;
}

Here you can see that we set the table rows (tr) to be #cdcdcd, and then set a pseudo-class later in the order for every other (starting at the second) cell to be a darker grey. You can use first-child or last-child for the first or last element of its type, however nth-child selects elements based on a simple algebraic expression which therefore select odd or even elements. In this case, 2n denotes even elements (2nd, 4th, 6th etc). There is another way of doing the process above, which is to use two separate sets of nth-child selectors, with one indicating one colour on even elements (2n), and the other indication another colour on the odd elements (2n-1). However, it makes sense to do it the way that we’ve set it, as it’s simple to order in the CSS.

I’ve also shown the selector for the row border, which uses the not: selector, indicating that the border should apply to every cell but the bottom table row. I’ll be playing with these as we go along, applying them to other selectors now that I’ve understood the format.

Navigation

Everyone needs a bit of direction, so I decided it was time to sort out my navigation! The website you see before you is the most up-to-date version, and therefore you’ll notice a few things that we fixed in later lessons. However, one of the main problems with navigation was the reponsiveness of my website after adding it.

I aligned my navigation by setting a left margin in order to push the list items to the right. Setting this margin in pixels was largely guesswork, which was eventually problematic. I used the position pseudo-class ‘li:first-child’ in order for this margin only to affect the first list item. This meant that only the first item would have a larger margin, pushing the others along with it. However, this broke my site, and made it look worse than it does on mobile! We’re going to fix this next week, as I don’t want to embarrass myself in front of all my new RWD friends.

I also created a list item class ‘.current’ to use within navigation to clearly indicate with an underline what section the reader was in, as I’ve created a new section where I talk about the other love in my life, Art. I’d like to revise this once I’ve looked a little harder at Design.

A note on Rubber Ducks, and how to find solutions

Aral has instilled within me the motto ‘Think about the solution first, then find the code’. As coding is pretty much a completely new thing to me, I’m not always going to instinctively know the answer. However, I will be able to demonstrate how I’d like to get there, or what I’d like to do. Also, I’ve learnt there’s no harm in trying, or failing, because it just means you can try something else.

One tip that friend and super-technologist Natalia Buckley passed on, was that of Rubber Ducking. It’s a good way of finding solutions by making yourself think out loud. Essentially you have a Rubber Duck, or whatever small mascot you can find, and you talk your problems through with them. I don’t mean problems like what to have for lunch, rather for asking questions like ‘Why won’t my navigation headers stay where I want them to?’. Then you explain what you’ve tried, and where you could go next, then try it, and ask again if it fails. Natalia mentioned that often you’ll unconsciously find the solution because you hadn’t considered something. This is working for me, however at the moment I don’t have a rubber duck, just a rather bemused pair of cats.
